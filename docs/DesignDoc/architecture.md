# 次世代要求定義パイプライン アーキテクチャ設計書

## 1. システムの目的とビジョン
本システムは、「正しいもの（Validatable）を正しく（Verifiable）作る」ことを目的とした次世代要求定義パイプラインである。
人間の「情熱と意図」を起点とし、USDM、BDD（Gherkin）、EARSを統合した4層のデータモデルを用いて要件を構造化する。機械的なLinterとLLM（大規模言語モデル）を組み合わせることで、上流フェーズにおける「意味論的な矛盾」と「状態遷移のバグ」を徹底的に排除する。

## 2. コア・データモデル（4層要求ツリー構造）
V&V（妥当性確認と検証）の関心事を分離し、フラット・リレーショナルなツリー構造としてモデリングする。

| レイヤー | 名称 | ドメイン | 構文・制約 | 役割と接続先 |
| :--- | :--- | :--- | :--- | :--- |
| **Layer 1** | **理由 (Why)** | Validation | 自然言語 | ユーザーやビジネスの「情熱・意図・目的」。最終的なビジネス価値の達成基準。 |
| **Layer 2** | **親要求 (Epic)** | Scope / Flow | 非定型可 | 業務フローや機能の全体像（スコープ）の括り。具体的な仕様は持たない。 |
| **Layer 3** | **子要求 (Scenario)** | Macro-Verification | Gherkin構文<br>(1動詞制約) | ユーザー視点の独立した振る舞い。E2Eテストや受け入れテストへ直結。 |
| **Layer 4** | **仕様 (Rule)** | Micro-Verification | EARS構文 | 条件分岐、異常系、エッジケースのカプセル化。単体/プロパティベーステストへ直結。 |

## 3. 状態管理と機能間干渉の解決メカニズム
大規模システム（ADAS等）における状態の組み合わせ爆発（State Explosion）を防ぐため、ドメイン駆動設計（DDD）とMBSEの思想を導入する。

* **階層化Glossary（用語集）による状態管理:**
  * **システム階層 `[SYS:*]`**: 車両全体・システム全体で共有されるグローバルステート（例: `[SYS:Emergency]`）。
  * **ソフトウェア階層 `[SW:Context:*]`**: Bounded Context内に閉じたローカルステート（例: `[SW:LKAS:Active]`）。
* **調停（Arbitration）ノードの分離:**
  * 機能間の競合は、各機能の仕様内に「隠れた優先度」として記述しない。
  * 独立した「調停要求ノード」を設け、周辺状況をEARSのガード条件として明示することで優先順位を決定する。

## 4. システムコンポーネントと役割分担 (AI-Driven Development)
人間の意図とAIの検証能力を最大化するレイヤードアーキテクチャを採用する。

1. **人間 (Domain Expert / PO)**
   * 「意図（Validation）」の設計。システムのゴール、情熱、要求の枠組みを定義する。
2. **C# Linter (静的解析 / Verification)**
   * EARS構文の逸脱、孤児ノード、用語のスコープ違反（他ドメインのローカルタグの直接参照等）をミリ秒単位で決定論的に弾く。
3. **Python LLM (意味論的検証 / Validation Review)**
   * 自己注意機構を活かし、Linterを通過した構造化データに対して「条件網羅の漏れ」や「仕様と理由（目的）のギャップ」を非同期で分析・指摘する。
