# 要求定義・開発運用ルール

本ドキュメントは、要求定義パイプラインを利用・開発・運用する際の絶対的なルールを定義する。

## 1. 記述と分割のルール
* **情熱の明文化**: `Layer 1: 理由` には、システムを開発する真の目的やビジネス価値を、システマチックな定型文に押し込めず、生の自然言語で記述すること。
* **1要求＝1動詞の原則**: `Layer 3: 子要求` は、Gherkin構文（Given / When / Then）を用い、必ず1つの動詞（アトミックなアクション）に限定して分割すること。
* **複雑性のカプセル化**: 状態遷移の条件分岐や例外処理などのミクロな制約は、`Layer 3` に記述せず、必ず `Layer 4: 仕様` のEARS構文（WHILE... IF... THEN THE SYSTEM SHALL...）内に隠蔽すること。

## 2. 状態（State）とGlossaryの運用ルール
* **ネームスペースの厳守**: 状態やイベントを記述する際は、必ずGlossaryに登録された完全修飾名（例: `[SYS:StateName]` または `[SW:Domain:StateName]`）を使用すること。
* **境界侵犯の禁止**: 特定のソフトウェアドメイン（例: `SW:A`）の仕様内で、他のドメイン（例: `SW:B`）のローカルステートを直接参照・操作してはならない。必ずシステム階層（`SYS`）のイベントを介して通信すること。
* **状態爆発の回避**: マイクロステート（UIの微細な状態など）はGlossaryのグローバルなタグとして露出させず、各仕様（EARS）の内部に留めること。

## 3. 検証（V&V）のプロセスルール
* **Linterファースト**: C# Linterによる静的解析（Verification）エラーが存在する状態の要求データは、LLMによる意味論的検証（Validation）に回してはならない（Garbage In, Garbage Outの防止）。
* **LLMの役割の限定**: LLMには業務ドメインの「正解」を推測させるのではなく、入力されたテキスト間の「論理の飛躍（LogicGap）」と「例外考慮漏れ（MissingEdgeCase）」の指摘に専念させること。
* **Human-in-the-loop**: LLMの検証結果は絶対視せず、あくまで人間に対する「批判的レビューア」として扱い、最終的な妥当性確認と承認（Approved）は人間が行うこと。
